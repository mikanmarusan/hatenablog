# AI時代に自分の型はあるか

Claude Code をメインに、追加のレビュー目的で Codex を使って開発をしている。

Claude Code が最強、いや Codex がかなり良い、やっぱり Claude Code しかかたん——そういう議論を眺めていて思うのは、**ツールの優劣ではなく、自分の中に「型」があるかどうかの方が重要ではないか** ということだ。

この記事は「Claude Code を使っていないやつは損をしている」「2026年のベストプラクティスはこれ」「最高峰のベンチマーク！」みたいな記事ではないし、ノウハウを示す記事でもない。1か月前に SoTA だった話もすぐに古くなっていき強制的にアンラーニングが要求されていく中で、驚きを醸し出している暇もない。一人のエンジニアとしてAIへの向き合い方を示すログである。なお、持論を書いているのでコンテンツの正確性は保証していない。

## 不確実性コーンの話

不確実性コーン（Cone of Uncertainty）という概念がある。ソフトウェアプロジェクトの初期段階ほど見積もりの誤差が大きく、進行につれて精度が上がっていくという概念だ。プロジェクトの初期段階ではコストやスケジュールの見積もりに4倍ものブレが生じうるが、フェーズが進むたびに不確実性が減り見積もり精度が上がっていくという経験則だ。多くの人が一度はみたことが見聞きしたものだと思う。

[f:id:mikanmarusan:20260409181744p:plain]

前職で 認証・認可や決済といったミッションクリティカルなシステムに長年関わってきた。さらに銀行やクレジットカードのシステムも見てきた。これらのシステム開発で最も怖いのは「不確実性」だった。大規模な開発投資をしているにもかかわらず不確実性が残っている。その破壊力は凄まじい。

結果として、設計フェーズの重要性は身に沁みている。主語は大きいが、**ソフトウェアの質は設計（以降、計画）でほぼ決定づけられる** と信じている。

## 計画に労力をかける

AI によりコーディングは桁違いのスピードを得ることができた。ただ、プロダクト開発・ソフトウェア開発全体のフェーズでいえば、コーディングはごく一部にすぎない。そして前述のとおり、ソフトウェアの質は設計でほぼ決定づけられると思っている。だから設計に労力をかけるべきだ。AI時代でいえば、コーディングよりは、**「実装計画（planning）」** にAIの力を費やすべきだ。

しかし、AI のplanningにはブレが生じやすい。同じことを複数回やっても同じplanningにならないことはとても多い。これはAIの2つの性質に起因すると考えている。

#### AI の非決定性（Non-determinism）

生成 AI は次のトークン候補の確率分布を計算し、ランダム性（temperature）を加味してサンプリングするモデルだ。自己回帰モデル特有の性質（トークン N の選択がトークン N+1 以降の全ての条件付き確率分布に影響する）も相まって、**序盤の小さな変動が最終結果を大きく左右** する。

#### AIにおける計画（planning）が向き合う解空間の広さ

- **構造的な制約が少ない**:  コーディングは型・構文・アルゴリズムといった制約により解が収束しやすいが、自然言語による planning は制約が弱い。
- **多峰性**:  広大な解空間の中に複数の妥当なpeak（局所解）が存在し、リクエストごとに別のpeakにたどり着くことがある。
- **重力の弱さ**:  人間は良くも悪くも最初の設計判断や過去の成功・失敗体験に引きずられる。結果として解空間が限定される「重力」がある。AI にはそれがない。

この2つの性質があるからこそ、planning に**型**を作ることに意味がある。planningにおける非決定性と解空間の広さに対して、自分の設計の経験と意思決定の軸を構造化しAI に渡す。そうすることで、planning時のAI の出力のブレを抑え、自分たちが意図した設計空間の中で計画を練ることができる。スキルやサブエージェントの設計も、この「型をどう上手に AI に伝えるか」の具体的な手段にすぎないとも言える。

Claude CodeでのHowとして、`/x-planning` というskillsを作って運用している。いつもの設計の「型」を実行するものである。skills 名に `x-`のprefixをつけているのは、built-inコマンドなのか自作なのかを区別するための自分なりの「型」の一つである。実際は英語で書かれているが、わかりやすさのために日本語に翻訳したものを下記に記す。


````
---
context: fork
allowed-tools: Agent, AskUserQuestion, Read(*), Glob(*), Grep(*), Write(./.plans/**), WebSearch, EnterPlanMode, ExitPlanMode
description: "Create an execution plan with 6 specialized Agent Teams"
---

# エージェントチームによる実行計画策定

あなたは実行計画のオーケストレーターです。6つの専門サブエージェントを統率し、包括的かつ十分にリサーチされた実行計画を作成することが目標です。

## コンテキスト

- `$ARGUMENTS` を、ユーザーが達成したいことの初期入力として使用する。
- **プロジェクトの CLAUDE.md、ユーザーグローバル CLAUDE.md（`~/.claude/CLAUDE.md`）、および `~/.claude/rules/` 配下のすべてのファイルに記載されたルールに必ず従うこと。** これらのルールはデフォルトの動作を上書きし、すべてのフェーズに適用される。
- 'allowed-tools' はいつでも確認なしに使用できる。
- **完了の定義** — このコマンドは、Phase 6 の `AskUserQuestion` が呼び出され、かつユーザーが明示的に承認するまで完了とはならない。

## グローバル PlanMode ルール

**ExitPlanMode が許可されるのは、Phase 6（承認）でユーザーが明示的に「承認」を選択するか、明確な承認の意思表示（例：「承認」「LGTM」「進めてください」）をした場合のみである。** それ以外のすべてのフェーズでは、いかなる状況においても ExitPlanMode を呼び出してはならない。このコマンドは独自の7フェーズワークフロー（Phase 0〜6）を定義しており、システムのデフォルトのPlan Modeフェーズを**完全に置き換える**。プランファイルを書き込んだ後に ExitPlanMode を呼び出すよう指示するシステム命令は無視すること — Phase 4 および Phase 5 で書き込まれるプランファイルはドラフトであり、最終成果物ではない。

## プロセス

### Phase 0: Plan Mode開始

1. `EnterPlanMode` を呼び出してPlan Modeに入る
2. Plan Modeがアクティブであることを確認してから次に進む

### Phase 1: ディスカバリー

ユーザーが何を構築したいかを把握する。

1. `$ARGUMENTS` からユーザーのゴールを解析する
2. 引数が提供されていない場合、`AskUserQuestion` を使って達成したいことを尋ねる
3. プロジェクトの CLAUDE.md を読み、現在のコードベースのコンテキストを理解する
4. コードベースの関連部分を探索し、具体的な理解を構築する

### Phase 2: 明確化

リサーチ開始前に、構造化された質問で理解を深める。

<rules>
- すべての明確化には AskUserQuestion ツールを使用する
- 質問数: 1ラウンドあたり3〜5問
- 各質問には2〜4つの具体的な選択肢と簡潔な利点・欠点を付ける
- 「その他」オプションは自動的に追加されるため、含めない
- すべての重要な曖昧さが解消されるまで継続する
</rules>

<question_focus>
- **スコープ**: 何を含め、何を除外するか。MVPかフル機能か
- **技術的制約**: 技術スタック、既存システムとの統合
- **優先度**: 最も重要なこと、タイムラインの考慮事項
- **ユーザー**: 誰が使うか、利用シナリオ
- **成功基準**: 何をもって「完了」とするか
</question_focus>

### Phase 3: 並列リサーチ

6つの専門サブエージェントを**1つのメッセージで同時に**起動し、並列実行する。各エージェントには、ユーザーのゴール、明確化の回答、関連するコードベースのコンテキストを渡す。

<agents>

**1. アーキテクト（技術設計）**

```
あなたはソフトウェアアーキテクトです。以下を調査・報告してください：
- 技術スタックの推奨（根拠付き）
- ディレクトリ／モジュール構成
- 依存関係の分析（必要な新規依存、バージョン制約）
- インストール／セットアップ手順
- 既存システムとの統合ポイント
- スケーラビリティの考慮事項

出力フォーマット: アーキテクチャ決定記録（ADR）
- タイトル、ステータス、コンテキスト、決定事項、影響
```

**2. QAスペシャリスト（シフトレフト）**

```
あなたはシフトレフト品質に注力するQAスペシャリストです。以下を調査・報告してください：
- エッジケースと障害モード
- テスト戦略（ユニット／統合／E2Eの内訳）
- 冪等性の要件
- データバリデーションの懸念事項
- 必要なエラーハンドリングパターン
- リグレッションリスクの領域

出力フォーマット: リスクマトリクス ＋ テスト戦略
- 発生可能性／影響度を評価したリスク項目
- コンポーネントごとの推奨テストカバレッジ
```

**3. セキュリティスペシャリスト（脅威モデリング）**

```
あなたはセキュリティスペシャリストです。以下を調査・報告してください：
- 提案システムのSTRIDE脅威モデル
- 認証／認可の要件
- データ保護の必要性（保存時、転送時）
- 設定ファイル変更のリスク（settings.json、.env など）
- サプライチェーンリスク（新規依存関係）
- プロダクトセキュリティの考慮事項

出力フォーマット: 脅威評価レポート
- リスク評価付き脅威マトリクス
- 推奨される緩和策
```

**4. プロフェッショナルプログラマー（開発）**

```
あなたはプロフェッショナルプログラマーです。以下を調査・報告してください：
- 実装アプローチ（ステップバイステップ）
- 準拠すべき既存のコードパターン
- 変更が必要なファイル（具体的な場所付き）
- API設計（該当する場合）
- 技術的負債の考慮事項
- コード再利用の機会

出力フォーマット: 実装ブループリント
- ファイルパス付きの順序付きタスクリスト
- コードパターンの参照
```

**5. UXデザイナー（体験設計）**

```
あなたはUXデザイナーです。以下を調査・報告してください：
- ユーザーフロー（プライマリおよびセカンダリ）
- インターフェース設計の考慮事項（CLI、Web、モバイルなど該当するもの）
- アクセシビリティ要件
- エラー状態とユーザーフィードバックの設計
- 段階的開示の戦略
- プロジェクトにUIコンポーネントがない場合、簡潔な理由とともに「該当なし」と報告する

出力フォーマット: UX仕様
- ユーザーフロー図（テキストベース）
- インタラクションパターン
```

**6. マーケットリサーチャー（競合分析）**

```
あなたはマーケットリサーチャーです。以下を調査・報告してください：
- 類似・競合するツールやアプローチ
- 業界のベストプラクティス
- 機能比較マトリクス
- 差別化の機会
- 既存ソリューションからの教訓

出力フォーマット: 市場分析レポート
- 競合比較表
- 主要な知見と推奨事項
```

</agents>

### Phase 4: プランドラフト

6つのエージェントレポートを統合し、プランファイルを作成する。

**統合ステップ：**

1. 各エージェントのレポートをレビューする
2. レポート間の矛盾を特定する（例：アーキテクトが X を推奨しているが、セキュリティが問題を指摘している）
3. 補完的な知見を特定する
4. 合理的な判断で矛盾を解決する

**プランファイルの作成：**

プランドキュメントを `.plans/` ディレクトリに書き込む。

**ファイル名**: `planning-{トピック}-{YYYYMMDD}.md`

<plan_template>

```markdown
# 計画: {トピック}

## エグゼクティブサマリー

[計画の概要を2〜3段落で記述]

## リサーチ結果

### アーキテクチャ
[アーキテクトエージェントからの統合結果]

### 品質保証
[QAスペシャリストエージェントからの統合結果]

### セキュリティ評価
[セキュリティスペシャリストエージェントからの統合結果]

### 実装ブループリント
[プロフェッショナルプログラマーエージェントからの統合結果]

### UX仕様
[UXデザイナーエージェントからの統合結果]

### 市場分析
[マーケットリサーチャーエージェントからの統合結果]

## 実装計画

### フェーズ
[順序付きフェーズと説明]

### タスク
[依存関係付きの詳細タスクリスト]
- [ ] タスク 1: ...
- [ ] タスク 2: ...

### マイルストーン
[主要なマイルストーンとチェックポイント]

## リスク評価
[全エージェントからの統合リスクと緩和策]

## Codex レビュー結果
[Phase 5 で記入]
```

</plan_template>

プランファイルの書き込み後、**直ちに Phase 5 に進む**。ExitPlanMode を呼び出してはならない。

### Phase 5: Codex レビュー

Codex CLI を直接呼び出し、プランドラフトを検証する。

**初回レビュー**（Phase 4 の後）：

1. エージェントを起動して Codex レビューを実行する：
   ```
   Agent({
     description: "Codex プランレビュー",
     prompt: "以下のコマンドを実行し、出力をそのまま返してください：
     codex exec --sandbox read-only --color never \"このプランをレビューして、現実的でクリティカルな点のみ指摘して: {plan_full_path} (ref: {CLAUDE.md full_path})\"
     codex がインストールされていない、または失敗した場合はエラーを報告してください。"
   })
   ```
2. エージェントの出力結果をプランファイルの `Codex レビュー結果` セクションに反映する

**再レビュー**（Phase 6 の「変更をリクエスト」から戻った場合）：

1. プランファイルの更新後、エージェントを起動して Codex セッションを再開する：
   ```
   Agent({
     description: "Codex プラン再レビュー",
     prompt: "以下のコマンドを実行し、出力をそのまま返してください：
     codex exec resume --last --sandbox read-only --color never \"プランを更新したのでレビューして、現実的でクリティカルな点だけ指摘して: {plan_full_path} (ref: {CLAUDE.md full_path})\"
     codex がインストールされていない、または失敗した場合はエラーを報告してください。"
   })
   ```
2. エージェントの出力結果をプランファイルの `Codex レビュー結果` セクションに反映する

**注意事項：**
- 一時ファイルは使用しない — プランファイルのパスをプロンプトに直接渡す
- `--sandbox read-only` で意図しないファイル変更を防止する
- `--color never` で ANSI エスケープコードが出力を汚染するのを防止する
- エージェントは実行時に `{plan_full_path}` と `{CLAUDE.md full_path}` を実際のパスに置換すること

プランファイルの更新後、**直ちに Phase 6 に進む**。ExitPlanMode を呼び出してはならない。

### Phase 6: 承認

ユーザーにプランを提示し、レビューを依頼する。

1. `Read` ツールを使って**プランファイルを読み込む**
2. **プランの全内容をテキストとして出力**し、ユーザーが会話内で直接確認できるようにする。このステップを省略してはならない — 出力しなければユーザーはプランファイルの内容を確認できない。
3. `AskUserQuestion` を以下の選択肢で使用する：
   - **承認**: プランを確定し、実装を開始する。
   - **変更をリクエスト**: 必要な修正を指定する。
   - **却下**: プランを破棄し、セッションを終了する。

**承認ゲート：**

- **ユーザーが明示的に「承認」を選択するか、明確に承認の意思表示（例：「承認」「LGTM」「実装に進んでください」）をしない限り、絶対に ExitPlanMode を呼び出してはならない。**
- コメント、質問、フィードバック、提案 — たとえポジティブなものであっても — 明示的な承認キーワードが含まれていない限り「変更をリクエスト」として扱う。
- 判断に迷う場合は、Plan Modeに留まり「変更をリクエスト」として対応する。

**承認時：**

1. `ExitPlanMode` を呼び出す
2. acceptEdits モードに切り替える
3. タスクセクションを上から順に実行して実装を開始する

**変更リクエスト時：**

1. Plan Modeに留まる（`ExitPlanMode` を呼び出さない）
2. リクエストされた修正でプランファイルを更新する
3. **再レビュー** : Phase 5 を再実行し、更新されたプランを検証する
4. 更新されたプランを再度読み込み、再提示する
5. 同じ3つの選択肢（承認／変更をリクエスト／却下）で `AskUserQuestion` を再度使用する

## 重要な注意事項

- **CLAUDE.md のルールは必須**: プロジェクトの CLAUDE.md、ユーザーグローバル CLAUDE.md、および `~/.claude/rules/` 配下のすべてのファイルに必ず従うこと。主要なルールには以下が含まれる：
  - **コード変更前に必ずフィーチャーブランチを作成する** — main に直接コミットしない
  - **コマンドのスコープ境界を遵守する** — このコマンドのスコープは計画策定のみ。ユーザーが明示的に承認しPlan Modeを終了しない限り、実装を行ってはならない
- 各サブエージェントのプロンプトには、完全なコンテキストとしてユーザーのゴールと明確化の回答を含めること
- サブエージェントはコードベースを探索（Read、Glob、Grep）し、必要に応じて WebSearch を使用すること
- サブエージェントの担当領域が該当しない場合（例：CLIのみのツールに対するUX）、無関係な内容を捏造するのではなく「該当なし」と報告すること
- プランドラフトは実行可能でなければならない — すべてのタスクに明確な検証基準を設けること
- ゴールが明確に見えても、明確化フェーズをスキップしない — 常に隠れた前提が存在する
````

一見すると Claude Code 公式の [feature-dev](https://github.com/anthropics/claude-plugins-official/tree/main/plugins/feature-dev) プラグイン と同じように見える。feature-dev は、Explorer → Architect → Reviewerの3エージェントによって探索・設計・レビューを一気通貫で動かすcustom slash commands/skillsだ。「どう作るか」に対してはびっくりするほど強い。

ただ、使い込むうちに一発で開発が完了しないことが多いことにも気づいた。計画が破綻するのは設計がまずいからではなく、設計の前提が間違っていたときだ。重大なセキュリティ要件があるとか、他のライブラリで実装済みであるとか。こうした問題は、開発工程の中からは見えないこともある。x-planning では Architect、QA、Security、Programmer、UX、Market Researcher の6つの専門職エージェントを並列に走らせている。設計の幅を横に広げた多角的な検証をすることで設計の前提が間違っていないかの不確実性を減らす。さらに作り上げた計画に対して、Codex CLI という完全に独立した外部AIにレビューさせる。feature-dev にとって計画は実装への通過点だが、x-planning にとっては計画そのものが成果物になる。言い換えれば、feature-dev は「正しく作る」仕組み、x-planning は「正しいものを作る」仕組みかもしれない。x-planning で計画を作り、そのplanファイルを元に feature-dev で実装する、という組み合わせもできる。

## 積み上げてきた技術の上で毎日向き合う

「プロンプト1つでここまできる！」「エンジニアはオワコン！」という X のポストもたくさん見かけるようになった。

先日の [開発生産性Conference 2025](https://findy-tools.io/articles/raksul-dev-productivity-con-2025/134) でも話したが、AI エージェントコーディングは急に登場したものではなく、ソフトウェア開発のベストプラクティスの上に成り立っている開発手法だ。

AI が作るコードの正しさはテストによって保証され、AI への学習は既存ドキュメントを使用し、学んだことを再度ドキュメント化することで複雑性を管理する。AI 自身がソースコードを書いて検証するためには標準化されたインフラ環境が必要だ。これら全ての要素が揃うことで開発が一直線に進む。

AI の技術の移り変わりは早い。ただ、もともと技術は移り変わりが激しく陳腐化しやすいものだった。それ自体は今に始まったことではない。
だからこそ技術者マインドは陳腐化させてはならない。AIをベースとした開発に毎日向き合おうと思っている。手を動かし、常に学ぶこと。これが自分を含む過去の技術者たちに対する最大のリスペクトかもしれないと思っている。下記、自分の草日記。

[f:id:mikanmarusan:20260409181748p:plain]

AI によってソースコードは書けるようになった。これからのエンジニアは、単なる実装者ではなく、ソフトウェア開発の構造を理解し自分ならどのような構造にすればもっとプロダクトを良く作れるか、生産性高く作れるか、ユーザーのために価値を作れるかを設計し、それを整えるエンジニアになることが求められる気がする。

ツールは変わる。モデルも変わる。自分の中の型を大事にして乗り越えたい。
